# Frontend-проект, реализующий vite-расширения для многостраничных php-сайтов

### Библиотеки/фреймворки:
- `vite` - сборщик
- `eslint` - линтер
- `prettier` - форматтер
- `typescript` - ЯП вместо классического JS для статической типизации
- `vue 3` - основной фреймворк для реализации реактивности
- `axios` - асинзронные ajax-запросы к серверу
- `inversify и reflect-metadata` - реализация DI и сервиса контейнеров, автоматическое связывание сервисов
- `i18n` - локализация и языковые фразы

### Линтер, форматер и editorconfig:
- Форматер и editorconfig - нужны для единого форматирования программного кода всеми разработчиками команды
- Линтер же используется для корректирования разработчика во время написния кода, для того чтобы минимизировать ошибки и избежать излишней реализации
- prod-сборка проекта невозможна если не код не будет соответствовать линтеру и форматеру

### Структура проекта:
- Проект использует классическую модульную ар-ру, где есть ядро или иной набор модулей проекта, которые переиспользуются во всех остальных модулях, и сами модули, которые могут независимо работать друг от друга
- В каждом модуле, есть readme-файл, описывающий краткую информацию о модуле, а так же если есть связанный модуль, то он тоже описывается там, чтобы его не утеряли при переносе
- Структура папки `/src` (пример текущего проекта):
  - `core` - (модуль) набор классов, функций и тд, который будет переиспользоваться во всех остальных модулях, без него работа приложения, невозможна
  - `ui` - (модуль) стили, ui-компоненты и тд, которые должны переиспользоваться во всех остальных модулях, если проект содержит ui-компоненты, необходимо перенести данный пакет
  - `modules` - (набор модулей) все остальные модули проекта, которые могут быть перенесены или не перенесены на другой проект, реализующие определённый функционал, заложенный разработчиком
  
Модуль внутри себя может содержать любую структуру, которую заложат разработчики, главное чтобы она была примерно одинаковой во всех модулях. Внедрять fsd, чистую или луковую архитектуру внутри модуля, на мой взгляд излишне для проектов, с жизненным циклом от нескольких месяцев до года (заказная разработка).

### Пример структуры модуля (самые часто используемые компоненты):
- `/enums` - Перечисления
- `/models` - DTO или иные объекты описывающие данные
- `/exceptions` - Объекты-исключения
- `/interfaces` - Интерфейсы
- `/service-ids` - Символьные идентификаторы для получения сервисов из контейнера (Интерфейс в ts удаляется при компиляции, поэтому используется такое промежуточное звено)
- `/utils` - Утилиты
- `/services` - Сервисы
- `/repositories` - Репозитории
- `/use-case` - Use-кейсы
- `/mappers` - Преобразователи формата данных
- `/factories` - Фабрики, билдеры
- `/translations` - Файлы с локализацией
- `/composable` - Композиция - функционал реализующий переиспользование логики внутри компонентов
- `/components` - Vue-компоненты или иные компоненты другого фреймворка
- `/entrypoint` - Точки входа - файлы для сборки расширений
- `types.ts` - Кастомные типы данных
- `*.d.ts` - Файлы с декларациями в typescript
- `constants.ts` - Константы модуля
- `di.ts` - Настройка DI для модуля
- `index.ts` - Экспортный файл для функционала модуля, для последующего переиспользования в других частях программного кода

### Использование расширений:
- Использование деления frontend'а на расширения позволяет использовать в проекте только то, что необходимо в данный момент странице, не подключая ненужный программный код
- Расширения указываются в файле `entrypoint-list.json` в виде массива, где указывается путь до файла в папке `/entrypoint` внутри модуля
- Монтирование:
  - dev-режим:
    - Создаётся файл `vite.hot`, в котором указывается URL локального node-js сервера, который будет отдавать файлы скриптов
    - Далее php-мост подключает соответствующий script-теги на страницу, которые вызывают скрипт `@vite/client`, который уже упраляет теми скриптами, которые требуется подключить на странице
    - Это позволяет использовать быстрый dev-режим для разработки, ради которого он и создавался
  - prod-режим:
    - Запускаются проверки линтера и форматера, для того чтобы проверить программный код
    - Скрипты, стили и тд собираются в папке `/dist`, где дополнительно создаётся файл-манифест, где указаны все правила, по которым php-мост должен подключать скрипты на странице
    - Далее php-мост читает манифест и подключает всё необходимое на странице при её загрузке
  - Публичная часть:
    - Расширения подключаются в публичной части подключаются с помощью php-моста `ViteFrontendBridge`, который расположен в `/local/modules/main.site/lib/Core/Providers/ViteFrontendBridge.php`
    - С помощью метода `registerEntry`, происходит добавления расширения, которое будет загружено при загрузке страницы, в качестве агрумента, используется строка `"название модуля"."название точки входа"` из файловой структуры frontend'а
    - С помощью отложенной функции php-мост генерирует теги, которые подключают расширения на страницу
  - Асинхронное монтирование расширений:
    - Реализуется с помощью синглтона `AsyncViteLoader`, который располагается в `/local/modules/main.site/assets/js/script.js` (файл исполняется напрямую браузером без предварительной сборки)
    - Данный скрипт позволяет асинхронно, на уже загруженной странице догружать расширения, как и в dev-режиме и в prod-режиме, настройки для скрипта определяются php-мостом
